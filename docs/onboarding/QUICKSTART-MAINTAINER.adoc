= Panoply — Quick Start for Maintainers
:toc:

== Read this first

The substantive maintainer education is the xref:../guide/for-maintainers.adoc[Maintainer
guide] — how the envelope discipline functions as a *review gate*, and how to
evaluate new projections, backends, and experimental features against it. This
page is the operational entry point.

== Honest status

Design phase: there is no release to package yet. "Maintaining" Panoply today
means *guarding the discipline* — keeping the charter, the docs, and the
machine-readable state honest and consistent — not packaging a binary.

== What you gate

* *Projections* — must elaborate accepted programs into Core + evidence + manifest,
  or reject them; claimed equivalences must be witnessed.
* *Backends* — must ship an explicit contract; the manifest is backend-relative.
* *Experimental features* — may not borrow neighbours' claims; reject / downgrade
  / mark outside-envelope.
* *The charter* — amend via ADR (`docs/decisions/`), never weaken in code.

== CI / governance gates

* Estate gates: `estate-rules`, `governance`, `openssf-compliance`,
  `lint-workflows`. Keep the `hyperpolymath/standards` reusable pins fresh
  (refresh to the current standards SHA when the staleness check flags them).
* Hypatia neurosymbolic scan is *advisory* here: the current findings are
  documented false positives on template scaffolding (see `.hypatia-ignore`) and
  the check is non-blocking. Treat a *new* finding in changed files as signal.

== Honesty ledgers (keep distinct)

* Implementation state — `.machine_readable/6a2/STATE.a2ml`,
  xref:../status/READINESS.adoc[READINESS].
* Metatheory state — xref:../status/PROOF-NEEDS.adoc[PROOF-NEEDS],
  xref:../status/PROOF-STATUS.adoc[PROOF-STATUS]. Closing an implementation hole
  is not proving soundness.

== Release

Gated by xref:../../AUDIT.adoc[AUDIT.adoc]. Before a release, confirm the
manifest story is real end-to-end — that accepted programs genuinely project,
carry evidence, and emit manifests.
